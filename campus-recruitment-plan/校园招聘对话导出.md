# 1. GitHub 项目选型与项目方向

## 用户

你去GitHub上帮我找找项目，我想用Java后端的技术栈攒一个自己的项目，拿去应付校招。

## 助手

我先按“校招可讲清楚 + Java 后端主流栈 + GitHub 可参考但能改造成个人项目”的标准筛一轮，不只找 star 高的。重点会放在 Spring Boot、MyBatis/JPA、Redis、MQ、权限、部署、压测/监控这些面试能聊的点。

我已经看到几个“能讲技术深度”的方向：通用后台脚手架、商城/秒杀、OJ、校园交易/招聘。我的判断会偏保守：校招项目最好别只是一套 CRUD 管理系统，得有一两个能把面试聊热的技术场景。

我帮你在 GitHub 上按“**校招可讲、Java 后端栈完整、不要一眼开源套壳**”筛了一轮。我的建议是：**别直接做商城/后台管理系统原样复刻**，太容易撞车；更适合做一个有校园业务场景的项目，然后吸收这些开源项目里的技术模块。

## 值得参考的 GitHub 项目

| 项目 | 适合你拿来学什么 | 我对校招可用性的判断 |
|---|---|---|
| **RuoYi-Vue-Plus** | 权限、租户、数据权限、Redis、MyBatis-Plus、Knife4j、Docker、分布式场景。它是基于 RuoYi-Vue 重写增强的框架，强调分布式集群和多租户。 | 适合当“底座”，但不要把它本身当简历项目。 |
| **ruoyi-vue-pro** | RBAC、数据权限、SaaS 多租户、工作流、支付、短信、商城、CRM、ERP、IM、AI、IoT 等模块都很全。 | 太大太全，适合拆模块学习，不建议整套照搬。 |
| **eladmin / eladmin-mp** | Spring Boot、Spring Security、JWT、Redis、Vue、RBAC、动态路由、数据字典、代码生成、日志、定时任务。 | 适合学习企业后台规范：异常处理、日志、权限、监控。 |
| **mall-learning / mall** | Spring Boot、MyBatis、Redis、RabbitMQ、Elasticsearch、MongoDB、MinIO、Docker，以及商品、订单、营销、会员等电商业务。 | 很经典，但校招撞车率高。建议只学订单、库存、搜索、延迟消息。 |

| **CampusHire / Campus-Trading** | 校园招聘、学生投递、企业发布岗位、校园二手交易、订单、聊天等业务场景。 | 业务场景贴近学生，但部分项目规模小，适合借 idea，不适合直接抄架构。 |

## 我最推荐你做的项目方向

**做一个“校园综合服务平台”，别叫管理系统。**  
比如：**校园兼职/实习招聘 + 简历投递 + 岗位推荐 + 在线沟通 + 抢名额/预约面试**。

这个方向比“商城”更像你自己的项目，也能自然塞进 Java 后端常见技术点：

| 模块 | 技术点 | 面试怎么讲 |
|---|---|---|
| 用户与权限 | Spring Security / Sa-Token、JWT、RBAC | 登录认证、角色权限、接口鉴权、按钮权限 |
| 岗位/活动发布 | Spring Boot、MyBatis-Plus、MySQL | 表设计、分页查询、条件搜索、状态流转 |
| 简历投递 | 事务、状态机、幂等 | 防止重复投递、投递状态变更一致性 |
| 岗位搜索 | Elasticsearch / MySQL Fulltext | 关键词搜索、筛选、排序、召回 |
| 热门岗位缓存 | Redis | 缓存穿透、缓存击穿、热点数据更新 |
| 面试预约/抢名额 | Redis + Lua / Redisson | 防超卖、分布式锁、库存扣减 |
| 消息通知 | RabbitMQ / RocketMQ | 异步通知、削峰、失败重试 |
| 文件上传 | MinIO | 简历、头像、企业资质文件存储 |
| 操作日志 | AOP + 注解 | 审计日志、异常日志、登录日志 |
| 部署 | Docker Compose、Nginx | 一键部署、环境隔离、前后端分离 |

## 最省力的搭建路线

你可以这样“攒”，但要攒得像自己的：

1. **底座参考**：RuoYi-Vue-Plus 或 eladmin-mp  
   只借鉴登录、权限、菜单、日志、代码结构，不要保留原始业务名。

2. **业务参考**：CampusHire / Campus-Trading  
   借校园招聘、投递、聊天、订单/状态流转这些业务灵感。

3. **技术亮点参考**：springboot-seckill 或 mall-learning  
   把“抢名额/预约面试”做成类秒杀场景，用 Redis + MQ + 限流把技术深度拉起来。

4. **加分项参考**：GlowXQ OJ  
   不是让你做 OJ，而是学习它的 Docker 部署、现代 Java/Spring Boot 3 技术栈和工程结构。

## 简历上可以这么写

**校园智能招聘与面试预约平台**  
基于 Spring Boot 3、MyBatis-Plus、MySQL、Redis、RabbitMQ、Elasticsearch、MinIO、Docker 构建的校园招聘平台，支持学生注册登录、简历上传、岗位搜索、在线投递、企业审核、面试预约、消息通知与后台权限管理。使用 Redis 缓存热门岗位并通过布隆过滤/空值缓存降低缓存穿透风险；针对热门岗位预约场景，基于 Redis Lua 脚本实现名额原子扣减，通过 RabbitMQ 异步创建预约记录并削峰；使用 AOP 记录操作日志，Docker Compose 完成 MySQL、Redis、MQ、MinIO 等服务的一键部署。

这段比“我做了一个商城系统”强多了，面试官也更容易顺着问：缓存、MQ、锁、事务、搜索、权限、部署——你有东西可聊。

## 我的最终建议

你不要从零硬写，也不要直接套 RuoYi 改标题。最稳路线是：

**RuoYi-Vue-Plus / eladmin-mp 做工程底座 + 校园招聘业务 + 秒杀预约模块 + Redis/MQ/搜索/MinIO/Docker 亮点。**

这样项目既不会太空，也不会一眼“开源换皮”。校招项目嘛，核心不是多大，而是你能把其中两三个技术点讲得像真踩过坑。

# 2. 技术点与模块表格拆解

## 用户

把用的技术点，与模块详细地用表格搞出来

## 助手

可以，下面我按一个**“校园智能招聘与面试预约平台”**来拆。这个项目适合 Java 后端校招，因为它不是纯 CRUD，又能自然塞进 Redis、MQ、权限、搜索、文件存储、定时任务、Docker 部署这些面试常问点。

参考思路可以借鉴几类开源项目：RuoYi-Vue-Plus 这类后台底座适合学权限、MyBatis-Plus、接口文档、工程结构；mall-learning 适合学 Redis、RabbitMQ、Elasticsearch、MinIO、Docker 等 Java 后端常见中间件；GlowXQ OJ 这类项目适合学 Spring Boot 3、Java 21、Docker 一键部署和更现代的工程组织；eladmin 适合学 Spring Security、JWT、Redis、前后端分离后台系统。

## 一、项目整体技术栈表

| 层级 | 技术点 | 用在项目哪里 | 校招面试可讲点 | 优先级 |
|---|---|---|---|---|
| 后端框架 | Spring Boot 3.x | 整个后端服务基础 | 自动配置、分层架构、配置管理、异常处理 | 必做 |
| 编程语言 | Java 17 / Java 21 | 后端主语言 | Stream、Optional、CompletableFuture、JVM 基础 | 必做 |
| ORM | MyBatis-Plus | 用户、岗位、投递、预约等表的 CRUD | 分页插件、条件构造器、逻辑删除、乐观锁 | 必做 |
| 数据库 | MySQL 8 | 核心业务数据存储 | 表设计、索引优化、事务隔离级别、慢 SQL | 必做 |
| 缓存 | Redis | 热门岗位、验证码、登录态、预约库存 | 缓存穿透、缓存击穿、雪崩、热点数据 | 必做 |
| 权限认证 | Spring Security / Sa-Token + JWT | 登录、角色、接口鉴权 | Token 登录、RBAC、权限注解、无状态认证 | 必做 |
| 消息队列 | RabbitMQ / RocketMQ | 投递通知、预约成功通知、异步创建预约单 | 削峰、异步解耦、消息可靠性、死信队列 | 强烈建议 |
| 搜索 | Elasticsearch / Meilisearch / MySQL Fulltext | 岗位关键词搜索 | 倒排索引、关键词分词、搜索排序 | 建议 |
| 文件存储 | MinIO | 简历、头像、企业资质上传 | 对象存储、预签名 URL、文件访问权限 | 建议 |
| 接口文档 | Knife4j / Swagger | 后端接口调试和文档 | RESTful API、接口分组、请求参数校验 | 必做 |
| 参数校验 | Hibernate Validator | 注册、投递、发布岗位等表单校验 | 注解校验、全局异常处理 | 必做 |
| 全局异常 | 自定义 BizException + ControllerAdvice | 统一错误返回 | 错误码设计、业务异常和系统异常区分 | 必做 |
| 日志 | Logback + AOP | 登录日志、操作日志、异常日志 | AOP 切面、审计日志、链路排查 | 必做 |
| 定时任务 | XXL-JOB / Spring Task | 过期岗位下线、预约超时取消 | 定时扫描、任务幂等、分布式任务 | 建议 |
| 限流 | Redis + Lua / Sentinel / Guava RateLimiter | 热门岗位预约接口 | 防刷、接口限流、令牌桶/滑动窗口 | 加分 |
| 分布式锁 | Redisson / Redis Lua | 面试名额扣减、防重复预约 | 锁粒度、锁续期、原子操作 | 加分 |
| 部署 | Docker Compose | MySQL、Redis、MQ、MinIO、后端服务部署 | 环境隔离、一键部署、容器网络 | 必做 |
| 网关/反代 | Nginx | 前端静态资源、后端接口代理 | 跨域、反向代理、负载均衡基础 | 建议 |
| 监控 | Spring Boot Actuator | 健康检查、运行状态 | 服务健康、接口耗时、JVM 指标 | 加分 |
| 测试 | JUnit + Postman + JMeter | 单测、接口测试、压测 | 压测报告、并发预约、防超卖验证 | 加分 |

## 三、核心模块与技术点对应表

| 核心技术点 | 最适合落在哪个模块 | 具体实现方式 | 面试官可能怎么问 | 你应该准备的回答方向 |
|---|---|---|---|---|
| JWT 登录 | 用户模块 | 登录成功生成 Access Token，Redis 存储用户权限信息 | Token 为什么放前端？怎么防伪造？ | 讲签名、过期时间、HTTPS、Redis 黑名单 |
| RBAC 权限 | 权限模块 | 用户-角色-菜单-权限四张表 | 权限表怎么设计？ | 讲角色授权、菜单权限、按钮权限、接口权限 |
| Redis 缓存 | 岗位模块 | 缓存热门岗位、岗位详情、字典数据 | 缓存和数据库不一致怎么办？ | 讲更新数据库后删缓存、延迟双删、TTL |
| 缓存穿透 | 岗位详情 | 查询不存在岗位时缓存空值 | 大量查不存在 ID 怎么办？ | 讲空值缓存、布隆过滤器 |
| 缓存击穿 | 热门岗位 | 热点岗位过期时加互斥锁 | 热点 Key 失效怎么办？ | 讲互斥锁、逻辑过期 |
| Redis ZSet | 浏览记录/排行榜 | 用 score 记录时间或热度 | 怎么做热门岗位排行？ | 讲 ZSet score、定时衰减、分页 |
| MQ 异步 | 消息通知模块 | 投递/预约成功后发送消息到 MQ | 为什么不用同步发通知？ | 讲削峰、解耦、失败重试 |
| MQ 幂等 | 预约通知 | messageId 唯一去重 | 消息重复消费怎么办？ | 讲唯一键、消费记录表、幂等更新 |
| Redis Lua | 面试预约模块 | Lua 脚本原子判断库存并扣减 | 怎么防止超卖？ | 讲 Redis 单线程 + Lua 原子性 |
| 分布式锁 | 预约/投递模块 | Redisson 锁住岗位 ID 或场次 ID | 锁失效怎么办？ | 讲看门狗、锁粒度、finally 释放 |
| MySQL 事务 | 投递模块 | 投递记录和状态日志同事务提交 | 事务什么时候会失效？ | 讲 Spring 事务传播、自调用失效、异常回滚 |
| 唯一索引 | 投递/收藏/预约 | user_id + job_id 唯一 | 怎么防重复投递？ | 讲数据库兜底 + 业务层幂等 |
| Elasticsearch | 岗位搜索 | job 表同步到 ES，支持关键词检索 | 数据库和 ES 不一致怎么办？ | 讲 MQ 同步、定时补偿、最终一致性 |
| MinIO | 简历模块 | 上传简历到对象存储，数据库保存 URL | 为什么不用直接存数据库？ | 讲文件大、数据库膨胀、对象存储更适合 |
| AOP 日志 | 操作日志模块 | 自定义 @OperationLog 注解 | 日志怎么自动记录？ | 讲切面、注解、反射、请求上下文 |
| 全局异常 | 所有模块 | @RestControllerAdvice 统一返回 | 为什么要统一异常？ | 讲前后端约定、错误码、排查效率 |
| 参数校验 | 表单提交 | @Valid + 注解校验 | 参数校验放前端还是后端？ | 讲前端体验、后端兜底 |

## 九、并发与一致性技术点

| 场景 | 技术点 | 解决的问题 | 面试表达 |
|---|---|---|---|
| 重复投递 | 唯一索引 + 业务校验 | 防止同一学生重复投同一岗位 | 业务层先查，数据库唯一索引兜底 |
| 重复预约 | 唯一索引 + Redis 去重 | 防止重复预约同一场次 | 先 Redis 快速判断，再 DB 兜底 |
| 名额超卖 | Redis Lua | 多人同时抢同一面试名额 | Lua 保证判断库存和扣减原子性 |
| 数据库扣减 | 乐观锁 / 条件更新 | 防并发覆盖 | `where remain_count > 0` |
| MQ 重复消费 | message_id 去重 | 防重复站内信 | 消费记录表或唯一索引 |
| 缓存不一致 | 删除缓存 + TTL + 补偿 | DB 和 Redis 不一致 | 先更新 DB，再删缓存 |
| ES 不一致 | MQ 同步 + 定时补偿 | job 表与 ES 索引不一致 | 最终一致性 |
| 定时任务重复执行 | 分布式锁 + 幂等状态判断 | 多实例重复跑任务 | Redisson 锁或状态条件更新 |
| 文件脏数据 | 定时清理 | 上传后业务未绑定 | 文件元数据状态 + 定时清理 |
| 接口刷请求 | 限流 | 防恶意预约/投递 | Redis 计数器、滑动窗口、令牌桶 |

## 十、工程化技术点

| 技术点 | 具体技术 | 用在项目哪里 | 必要程度 |
|---|---|---|---|
| 统一返回 | Result / R | 所有接口 | 必做 |
| 全局异常 | `@RestControllerAdvice` | 业务异常、参数异常 | 必做 |
| 错误码 | ErrorCode Enum | 前后端联调 | 必做 |
| DTO / VO 分层 | 入参 DTO、出参 VO | 所有模块 | 必做 |
| 枚举管理 | StatusEnum、UserTypeEnum | 状态字段 | 必做 |
| 常量管理 | RedisKey、MQConstant | Redis、MQ | 必做 |
| 自定义注解 | `@OperationLog` | 操作日志 | 建议 |
| 接口文档 | Knife4j / springdoc-openapi | API 调试 | 必做 |
| 多环境配置 | dev、test、prod | 部署配置 | 必做 |
| 配置脱敏 | 环境变量 | 密码、密钥 | 必做 |
| 日志规范 | Logback | 业务日志、异常日志 | 必做 |
| 健康检查 | Actuator | 服务状态 | 建议 |
| 代码生成 | MyBatis-Plus Generator | Entity、Mapper、Service | 可选 |

springdoc-openapi 可以基于 Spring Boot 项目自动生成 OpenAPI 文档；Spring Boot Actuator 则提供生产可用的健康检查和运行状态能力，校招项目里加这两个会显得工程完整度更高。

## 十一、部署与运维技术点

| 技术点 | 具体技术 | 用在项目哪里 | 必要程度 | 面试可讲点 |
|---|---|---|---|---|
| Docker | Dockerfile | 后端服务镜像 | 必做 | 镜像构建、分层 |
| Docker Compose | 编排 MySQL、Redis、RabbitMQ、MinIO、ES | 本地一键启动 | 必做 | 多容器编排 |
| Nginx | 反向代理、静态资源 | 前后端部署 | 建议 | 跨域、负载均衡基础 |
| 环境变量 | `.env` | 数据库密码、端口 | 必做 | 配置隔离 |
| 健康检查 | Actuator + Docker healthcheck | 服务可用性 | 建议 | 服务探活 |
| 日志挂载 | volume | 容器日志持久化 | 建议 | 问题排查 |
| 数据卷 | Docker volume | MySQL、Redis、MinIO 数据 | 必做 | 数据持久化 |
| CI/CD | GitHub Actions | 自动构建测试 | 加分 | 提交后自动检查 |

Docker Compose 官方定位就是用来定义并运行多容器应用，所以这个项目非常适合用一个 `docker-compose.yml` 把 MySQL、Redis、RabbitMQ、MinIO、Elasticsearch 和后端服务一起拉起来。

| 简历技术点 | 推荐写法 |

| Docker Compose | 使用 Docker Compose 编排 MySQL、Redis、RabbitMQ、MinIO、Elasticsearch 和后端服务，实现一键部署 |
| JMeter 压测 | 使用 JMeter 模拟高并发预约场景，验证系统在并发下无超卖问题 |
